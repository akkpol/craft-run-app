# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

This plan defines a staged rollout from a low-cost Drive-first storage workflow to a provider-based R2-backed workflow for FOGUS. The plan keeps Supabase as the source of truth for workflow and file metadata, allows the shop to continue working from Google Drive during the transition, and ensures customer-facing media delivery can move to controlled signed URLs later without reworking the core lead, quote, job, or admin flows.

## 1. Requirements & Constraints

- **REQ-001**: Supabase must remain the source of truth for operational metadata, workflow state, customer linkage, and file registry records throughout the rollout.
- **REQ-002**: The shop must be able to continue using Google Drive as its internal working/archive store during the early rollout phases.
- **REQ-003**: Customer-facing media access must not depend on raw Google Drive share links.
- **REQ-004**: The rollout must support both customer reference media and future production media without forcing a same-day migration of all historical files.
- **REQ-005**: Staff must be able to find the current file location from the admin surface without guessing whether the file lives in Drive, Supabase Storage, or R2.
- **SEC-001**: Google Drive credentials, Supabase secret keys, and future R2 credentials must remain server-only.
- **SEC-002**: Customer-accessible file URLs must be short-lived, permission-checked, and generated by application code instead of permanent third-party links.
- **SEC-003**: Files uploaded by customers through LINE or LIFF must be copied into application-controlled storage before their external source expires.
- **CON-001**: Preserve the current LIFF endpoint contract at `/liff` and the intake form entry at `/liff/intake`.
- **CON-002**: Preserve current workflow state behavior defined in `docs/workflow-policy.json`; this rollout changes storage behavior, not workflow transitions.
- **CON-003**: The current repo already contains provider metadata for `lead_media_assets` via `storage_provider` and `storage_bucket`; any Drive-first design must build on that direction rather than invent a parallel registry.
- **CON-004**: Google Drive is an operator-facing storage layer during early phases, not the long-term customer delivery layer.
- **GUD-001**: Implement storage-provider changes behind adapters or service-layer boundaries so UI components do not branch on provider-specific logic.
- **GUD-002**: Keep rollback simple: if any new provider path fails, the shop must still be able to open the working file from Drive while customer upload and admin metadata continue functioning.
- **PAT-001**: Prefer additive rollout with dual-write or mirror phases before provider cutover.
- **PAT-002**: Treat manual shop behavior as part of the rollout design; document where staff are expected to drag files into Drive and where automation takes over.

## 3. Alternatives

- **ALT-001**: Keep Google Drive as both the working store and the permanent customer-facing delivery layer. Rejected because Drive sharing semantics are not a reliable permission boundary for customer media and create long-term link hygiene problems.
- **ALT-002**: Migrate all Drive files to R2 in one cutover. Rejected because the shop needs continuity during production work and historical files do not all justify same-day migration.
- **ALT-003**: Keep using only Supabase Storage and skip Drive or R2 entirely. Rejected because the shop already operates with Drive-like working behavior and media-heavy usage can justify a cheaper delivery layer later.

## 4. Dependencies

- **DEP-001**: A Google Drive folder strategy agreed by the shop for active jobs, closed jobs, and archived jobs.
- **DEP-002**: Server-side Google Drive integration credentials or an explicit manual operator step if automation is delayed.
- **DEP-003**: Cloudflare R2 bucket, endpoint, and credentials for the later cutover phases.
- **DEP-004**: Supabase tables that can store provider metadata without losing compatibility with existing `supabase` and `r2` rows.
- **DEP-005**: Staff training for the dual-provider period so they know whether a given file should be opened from Drive or previewed from the app.

## 6. Testing

- **TEST-001**: Verify that provider-aware file rows can represent `google_drive`, `supabase`, and `r2` without breaking current customer-media tests.
- **TEST-002**: Manual-test staff open flows for Drive-resident files from the admin dashboard.
- **TEST-003**: Manual-test customer preview flows for application-controlled file-view routes without exposing raw Drive links.
- **TEST-004**: Validate that a file mirrored to R2 is served from R2 while the same record still retains a Drive fallback for staff during pilot rollout.
- **TEST-005**: Run a pilot with a small set of real jobs and confirm that lead creation, quote creation, admin preview, and staff file access all remain intact across the dual-provider window.

## 7. Risks & Assumptions

- **RISK-001**: Manual Drive handling can create metadata drift if staff move or rename files outside the expected folder structure.
- **RISK-002**: Google Drive preview and permission behavior may be inconsistent for customer-safe delivery, which is why Drive should not remain the customer-facing provider.
- **RISK-003**: Dual-provider periods increase support complexity unless the admin UI clearly shows which provider currently owns delivery versus archive.
- **RISK-004**: If archive policies are vague, the shop may keep duplicate copies across Drive and R2 longer than intended, increasing cleanup cost and confusion.
- **ASSUMPTION-001**: The shop is willing to keep using Drive for internal working files during the transition period.
- **ASSUMPTION-002**: Supabase remains online and authoritative for file metadata even when the underlying binary is stored elsewhere.
- **ASSUMPTION-003**: R2 is the intended final delivery store for active customer-facing media once rollout criteria are met.
