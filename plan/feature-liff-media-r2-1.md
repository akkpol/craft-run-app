# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

This plan prepares two related follow-up tracks for FOGUS: a low-risk media architecture that can take advantage of an existing Cloudflare R2 setup for frequent image upload/display, and a LIFF intake form UX refactor that uses less vertical space while staying safe inside the LINE webview. The goal is to improve media-heavy usage without breaking the current LIFF, webhook, quote, or admin preview flow.

## Role In Plan Stack

This file is a scoped feature plan, not the primary execution plan.

Use it only when the current task is specifically about frequent media upload/display, Cloudflare R2 evaluation, or LIFF intake form density and clarity improvements.

The overall delivery order and current wave priority still come from [process-go-live-waves-1.md](process-go-live-waves-1.md).

Deploy sequencing rule: if the customer LIFF intake UX is stable before the storage-provider work is ready, ship the LIFF-only improvement first. Do not hold a customer-facing LIFF deploy behind the R2 rollout track unless the release explicitly depends on the new media backend.

## 1. Requirements & Constraints

- **REQ-001**: Customer reference upload and preview must support frequent image-heavy usage without degrading the current lead, quote, and admin preview workflow.
- **REQ-002**: The LIFF customer form in `src/app/liff/intake/intake-form.tsx` must become denser and easier to complete on small screens without reducing clarity.
- **REQ-003**: The customer upload flow and production evidence flow must remain separate concerns even if they share storage abstractions.
- **REQ-004**: Admin must continue to see uploaded customer reference media through the existing backoffice snapshot/read model.
- **REQ-005**: The plan must preserve Thai-first copy and keep the form understandable with minimal screen usage.
- **SEC-001**: Cloudflare R2 credentials must remain server-only; do not expose access keys, secret keys, or raw bucket credentials to LIFF clients.
- **SEC-002**: Signed asset access must be generated by server-side code only and must remain time-limited.
- **CON-001**: Preserve the current LIFF endpoint contract at `/liff` and the intake form entry at `/liff/intake`.
- **CON-002**: Preserve `liff.init()` before `liff.requestFriendship()` and do not introduce LIFF flows that depend on unsupported browser APIs inside LINE.
- **CON-003**: Keep workflow state, lead creation, quote creation, and admin preview metadata in Supabase even if blob storage is abstracted.
- **CON-004**: Phase 1 must not require direct browser-to-R2 uploads; uploads should continue through the application boundary first to reduce LINE/LIFF risk.
- **CON-005**: LIFF UX and LIFF deploy sequencing must stay decoupled from the optional R2 rollout; the customer form may ship first on the current storage path if that is the fastest safe launch path.
- **GUD-001**: Introduce a storage adapter boundary before changing the active media backend.
- **GUD-002**: Keep a fallback path to the current Supabase Storage implementation until R2 behavior is validated end to end.
- **PAT-001**: Use progressive disclosure, compact previews, inline validation, and clearer hierarchy to reduce form height instead of hiding important fields without context.
- **PAT-002**: Treat LINE/LIFF regressions as launch-grade risks because they are expensive to debug after deployment.

## 3. Alternatives

- **ALT-001**: Replace Supabase Storage with direct browser-to-R2 uploads in one cut. Rejected because LIFF and LINE browser behavior are the highest-risk part of the stack and should not depend on a new direct-client upload contract first.
- **ALT-002**: Leave all media on Supabase Storage forever and only restyle the form. Rejected because the user already has R2 capacity and media-heavy usage may justify a provider abstraction and delivery optimization.
- **ALT-003**: Shrink the LIFF form only by reducing labels and copy. Rejected because the real issue is interaction density and clarity, not just fewer words.

## 4. Dependencies

- **DEP-001**: Cloudflare R2 bucket, access key, secret, and region/endpoint details.
- **DEP-002**: A final decision on whether R2 is private-only with signed access or fronted by a controlled CDN/domain.
- **DEP-003**: Supabase remains available for metadata tables, workflow state, and existing preview readers.
- **DEP-004**: LINE test accounts and at least one real mobile validation device.

## 6. Testing

- **TEST-001**: Unit-test customer media validation rules and provider adapter fallback behavior.
- **TEST-002**: Validate LIFF intake success and warning states when files upload successfully, partially fail, or are omitted.
- **TEST-003**: Manual-test upload preview and delete behavior inside LINE on a real mobile device.
- **TEST-004**: Manual-test admin preview retrieval for the same uploaded assets through the chosen provider.
- **TEST-005**: Manual-test repeated image retrieval to confirm signed access expiry and renewal behavior do not confuse customers or staff.

## 7. Risks & Assumptions

- **RISK-001**: A direct storage-provider switch without an adapter boundary can break both customer upload and admin preview at once.
- **RISK-002**: LINE webview behavior around large files, HEIC images, and cross-origin previews can differ from desktop browser behavior.
- **RISK-003**: Making the LIFF form visually smaller without improving hierarchy can increase submission mistakes.
- **RISK-004**: R2 signed-read semantics may require an application proxy or stricter URL lifecycle management than the current Supabase signed URL path.
- **ASSUMPTION-001**: Cloudflare R2 is already available and can be used as an infrastructure dependency when implementation begins.
- **ASSUMPTION-002**: Supabase remains the source of truth for operational metadata even if binary storage moves.
